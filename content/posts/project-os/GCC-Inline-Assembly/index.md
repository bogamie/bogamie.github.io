## 인라인 어셈블리 사용 이유

- OS 개발에서는 CR3 로드, 포트 I/O, 인터럽트 제어 등 C로 표현할 수 없는 CPU 명령어를 직접 사용해야 함
- 별도 `.S` 파일 없이 C 코드 안에서 어셈블리를 삽입할 수 있는 GCC 확장 기능이 인라인 어셈블리임

# 기본 문법

```c
asm [volatile] (
    "어셈블리 템플릿"
    : 출력 오퍼랜드
    : 입력 오퍼랜드
    : 클로버 리스트
);
```

- 총 4개의 영역이 콜론(`:`)으로 구분됨
- 출력, 입력, 클로버는 각각 생략 가능하지만 앞쪽을 건너뛰려면 빈 콜론을 써야 함

## 예시

```c
static inline void load_cr3(uint64_t pml4_phys) {
    asm volatile(
        "mov %0, %%cr3"
        :
        : "r"(pml4_phys)
        : "memory"
    );
}
```

| 영역 | 내용 | 의미 |
| --- | --- | --- |
| 템플릿 | `"mov %0, %%cr3"` | 입력 오퍼랜드 %0을 cr3에 대입 |
| 출력 | *(비어있음)* | 이 명령은 C 변수에 결과를 돌려주지 않음 |
| 입력 | `"r"(pml4_phys)` | pml4_phys를 범용 레지스터에 넣어 전달 |
| 클로버 | `"memory"` | 이 명령 전후로 메모리 내용이 바뀔 수 있음을 컴파일러에 알림 |

# 어셈블리 템플릿

```c
"mov %0, %%cr3"
```

- `%0`, `%1`, … : 오퍼랜드 참조 번호
    - 출력 오퍼랜드부터 0번 시작, 입력 오퍼랜드가 이어서 번호를 받음
    - 위 예시에서는 출력이 없으므로 `%0` = 첫 번째 입력 = `pml4_phys`
- `%%cr3` : 실제 레지스터 이름
    - 인라인 어셈블리에서는 `%`가 오퍼랜드 참조 문법이므로 실제 레지스터를 가리킬 때는 `%%`로 이스케이프해야 함
    - `%cr3` → 오퍼랜드 참조로 해석되어 오류 발생
    - `%%cr3` → AT&T 문법의 `%cr3`으로 올바르게 치환됨

# 오퍼랜드 제약 조건(Constraint)

- 제약 조건은 컴파일러에게 "이 C 변수를 어디에 배치하라"고 지시하는 문자임

## 자주 쓰는 제약 조건

| 제약 | 의미 |
| --- | --- |
| `"r"` | 범용 레지스터 아무거나 |
| `"a"` | `rax`/`eax` |
| `"b"` | `rbx`/`ebx` |
| `"c"` | `rcx`/`ecx` |
| `"d"` | `rdx`/`edx` |
| `"D"` | `rdi`/`edi` |
| `"S"` | `rsi`/`esi` |
| `"m"` | 메모리 주소 직접 참조 |
| `"i"` | 즉시값(상수) |
| `"0"`, `"1"`, … | N번 오퍼랜드와 같은 위치 사용 |

## 출력 제약 수식자

| 수식자 | 의미 |
| --- | --- |
| `"=r"` | write-only, 컴파일러가 아무 레지스터에 결과를 배치 |
| `"+r"` | read-write, 입력으로도 쓰고 출력으로도 씀 |

- 출력 오퍼랜드에는 반드시 `=` 또는 `+` 중 하나가 붙어야 함

## 예시: `"a"` 제약

```c
uint32_t low, high;
asm volatile(
    "rdtsc"
    : "=a"(low), "=d"(high)
);
```

- `rdtsc`는 결과를 `eax`, `edx`에 고정으로 넣는 명령어이므로 제약 조건도 `"a"`, `"d"`로 고정해야 함
- `%0` = `low` → `eax`, `%1` = `high` → `edx`

# 클로버 리스트

- 어셈블리 명령이 **컴파일러가 모르는 부작용**을 일으킬 때 명시하는 영역

## 자주 쓰는 클로버

| 클로버 | 의미 |
| --- | --- |
| `"memory"` | 메모리 내용이 변경될 수 있음 → 컴파일러가 메모리 접근 순서를 재배치하지 않음 |
| `"cc"` | 플래그 레지스터(EFLAGS)가 변경됨 |
| `"rax"` 등 | 해당 레지스터가 변경됨 → 컴파일러가 그 레지스터에 다른 값을 보관하지 않음 |

- `load_cr3`에서 `"memory"`를 쓰는 이유:
    - CR3를 바꾸면 페이지 테이블이 전환되어 모든 가상 메모리 매핑이 바뀜
    - 컴파일러가 CR3 변경 전후의 메모리 접근을 재배치하면 잘못된 주소를 참조할 수 있음
    - `"memory"` 클로버는 이 명령을 메모리 배리어로 만들어 재배치를 방지함

# volatile

```c
asm volatile( ... );
```

- `volatile`이 없으면 컴파일러는 출력이 사용되지 않는 asm문을 **제거하거나 이동**할 수 있음
- `volatile`을 붙이면 이 asm문은 반드시 이 위치에서 실행되어야 한다고 보장함
- OS 코드에서는 거의 항상 `volatile`을 붙임
    - CR 레지스터 조작, 포트 I/O, `hlt`, `cli`/`sti` 등은 부작용 자체가 목적이므로 제거되면 안 됨

# 여러 줄 어셈블리

```c
asm volatile(
    "cli\n\t"
    "mov %0, %%cr3\n\t"
    "sti"
    :
    : "r"(pml4_phys)
    : "memory"
);
```

- 각 명령어 사이에 `\n\t`를 넣어 줄바꿈 + 탭으로 구분함
- C 문자열 연결로 여러 줄을 이어 쓸 수 있음

# OS 개발에서 자주 쓰는 패턴 모음

## CR 레지스터 읽기

```c
static inline uint64_t read_cr3(void) {
    uint64_t val;
    asm volatile("mov %%cr3, %0" : "=r"(val));
    return val;
}
```

- 출력 제약 `"=r"` : 결과를 범용 레지스터에 넣어 `val`로 반환

## 포트 I/O

```c
static inline void outb(uint16_t port, uint8_t val) {
    asm volatile("outb %0, %1" : : "a"(val), "Nd"(port));
}
```

- `outb`는 `al`에서 `dx` 포트로 데이터를 보내는 명령어이므로 제약 조건을 `"a"`, `"Nd"`로 지정함
- `"N"` : 0~255 범위의 즉시값 허용(포트 번호가 상수일 때 `out` 즉시값 형태 사용 가능)

## 인터럽트 제어

```c
static inline void cli(void) { asm volatile("cli" ::: "cc"); }
static inline void sti(void) { asm volatile("sti" ::: "cc"); }
```

- 입출력 오퍼랜드 없이 클로버만 있을 때 빈 콜론 두 개(`:::`)로 건너뜀

## halt

```c
static inline void hlt(void) { asm volatile("hlt"); }
```

- 가장 단순한 형태, 오퍼랜드도 클로버도 없음
