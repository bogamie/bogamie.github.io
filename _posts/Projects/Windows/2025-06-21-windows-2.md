# 2.1 Overview of the Windows I/O Model
&nbsp;&nbsp; 모든 운영체제는 주변 장치와 데이터를 주고받는 흐름을 처리하기 위해 암시적 또는 명시적인 I/O 모델을 가지고 있다. Microsoft Windows I/O 모델의 한 가지 특징은 비동기 I/O를 지원한다는 것이다. 또한 I/O 모델은 다음과 같은 일반적인 특징들을 가지고 있다:
* I/O Manager는 최하위, 중간, 파일 시스템 드라이버를 포함한 모든 커널 모드 드라이버에게 일관된 인터페이스를 제공한다. 드라이버에 대한 모든 I/O 요청은 I/O 요청 패킷(IRP) 형태로 보내진다.
* I/O 작업은 계층적으로 이루어진다. I/O Manager는 I/O 시스템 서비스를 내보내고, 사용자 모드 보호 하위 시스템들은 애플리케이션이나 최종 사용자들을 대신하여 I/O 작업을 수행하기 위해 호출한다. I/O Manager는 이러한 호출을 가로채 하나 이상의 IRP를 생성하고, 계층적으로 구성된 드라이버를 통해 물리적 장치로 보낸다.
* I/O Manager는 드라이버가 지원할 수 있는 표준 루틴 집합을 정의한다. 이 중 일부는 필수적이고 일부는 선택 사항이다. 버스, 기능, 필터, 파일 시스템 드라이버 등 다양한 장치와 요구되는 기능의 차이가 있음에도 모든 드라이버는 비교적 일관된 구현 모델을 따른다.
* 운영체제 자체와 마찬가지로, 드라이버 역시 객체 기반이다. 드라이버, 그들이 제어하는 장치, 시스템 하드웨어 모두가 객체로 표현된다. I/O Manager와 기타 운영체제 구성 요소들은 드라이버가 적절한 객체를 조작하여 작업을 수행할 수 있도록 커널 모드 지원 루틴을 제공한다.

<div class="bg"></div>

&nbsp;&nbsp; IRP를 사용하여 전통적인 I/O 요청을 전달하는 것 외에도, I/O Manager는 PnP 및 전력 관리와 협력하여 PnP 및 전력 요청이 포함된 IRP를 보내는 역할도 수행한다.

# 2.2 End-User I/O Requests and File Objects
&nbsp;&nbsp; 커널 모드 드라이버는 Windows나 POSIX와 같이 잘 알려진 프로그래밍 인터페이스를 구현하는 보호된 하위 시스템에 의해 최종 사용자로부터 숨겨져 있다. 장치들은 사용자 모드 코드(보호된 하위 시스템도 포함함)에서 I/O Manager가 제어하는 이름이 지정된 파일 객체로만 볼 수 있다.

&nbsp;&nbsp; 아래 그림은 최종 사용자, 하위 시스템, I/O Manager 간의 관계를 나타낸다.
<img class="lazy invert" data-src="https://raw.githubusercontent.com/kisue01/assets/refs/heads/main/lmetystrmhver2dsxgaj.avif#center" alt="image" height="50%" width="50%" onclick="showImage(this)">

&nbsp;&nbsp; Win32 하위 시스템과 같은 보호된 하위 시스템은 I/O 요청을 I/O 시스템 서비스를 통해 적절한 커널 모드 드라이버에게 전달한다. 하위 시스템은 디스플레이, 비디오 어뎁터, 키보드, 마우스 장치 드라이버의 지원에 의존한다.

&nbsp;&nbsp; 보호된 하위 시스템은 최종 사용자와 애플리케이션이 드라이버를 포함한 커널 모드 구성 요소들에 대해 아무것도 알 필요가 없도록 보호한다. 그에 따라 I/O Manager는 보호된 하위 시스템이 하드웨어별 장치 구성이나 드라이버의 구현 방식에 대해 알 필요가 없도록 보호한다.

&nbsp;&nbsp; 또한 I/O Manager의 계층적 접근 방식은 대부분의 드라이버가 다음과 같은 사항에 대해 알 필요가 없도록 보호한다:
* I/O 요청이 Win32와 POSIX와 같은 특정 보호된 하위 시스템에서 발생했는지 여부
* 특정 보호된 하위 시스템이 어떤 종류의 유저 모드 드라이버를 가졌는지
* 각 보호된 하위 시스템이 사용하는 I/O 모델이나 드라이버와의 인터페이스가 무엇인지

<div class="bg"></div>

&nbsp;&nbsp; I/O Manager는 드라이버에게 단일 I/O 모델, I/O 작업을 수행할 수 있는 커널 모드 지원 루틴 집합, I/O 요청의 발생자와 그 요청에 응답해야 하는 드라이버 사이의 일관된 인터페이스를 제공한다.

&nbsp;&nbsp; 하위 시스템과 그 네이티브 애플리케이션은 I/O Manager가 제공하는 파일 객체 핸들을 통해서만 드라이버의 장치 또는 대용량 저장 장치의 파일에 접근할 수 있다. 이러한 파일 객체를 열거나 장치 또는 데이터 파일에 대한 핸들을 얻기 위해 하위 시스템은 요청된 파일을 열라는 요청과 함께 I/O 시스템 서비스를 호출한다. 요청된 파일은 파일 객체의 커널 모드 이름에 대한 하위 시스템별 별칭을 가질 수 있다.

&nbsp;&nbsp; 시스템 서비스를 내보내는 I/O Manager는 장치나 데이터 파일을 나타내는 파일 객체를 찾거나 생성하고, 절적한 드라이버를 찾는 역할을 담당한다.

# 2.3 The Life of an I/O Request
<img class="lazy invert" data-src="https://raw.githubusercontent.com/kisue01/assets/refs/heads/main/puzl4dpw84uccbg4t6tb.avif#center" alt="image" height="50%" width="50%" onclick="showImage(this)">

1. 하위 시스템은 이름이 지정된 파일을 열기 위해 I/O 시스템 서비스를 호출한다.
2. I/O Manager는 Object Manager를 호출하여 요청된 파일을 조회하고 해당 파일 객체에 대한 심볼릭 링크를 해석하는 데 도움을 받는다. 또한 보안 참조 모니터(Security Reference Monitor)를 호출하여 하위 시스템이 해당 파일 객체를 열 수 있는 적절한 접근 권한을 가지고 있는지 확인한다.
3. 만약 볼륨이 아직 마운트되지 않았다면, I/O Manager는 파일 열기 요청(open request)을 일시적으로 보류하고 하나 이상의 파일 시스템을 호출하여 그 중 하나가 해당 파일 객체를 자신이 사용하는 대용량 저장 장치 중 하나에 저장된 것으로 인식할 때까지 기다린다. 파일 시스템이 볼륨을 마운트하면 I/O Manager는 요청을 다시 진행한다.
4. I/O Manager는 열기 요청을 위해 IRP에 필요한 메모리를 할당하고 초기화한다. 드라이버 관점에서 열기는 "생성" 요청과 동등하게 취급된다.
5. I/O Manager는 파일 시스템 드라이버를 호출하면서 IRP를 전달한다. 파일 시스템 드라이버는 IRP의 I/O 스택 위치(I/O stack location)에 접근하여 어떤 작업을 수행해야 하는지 확인하고, 매개변수를 점검한 뒤, 요청된 파일 캐시에 있는지 확인한다. 만약 파일이 캐시에 없다면 IRP에서 다음 하위 드라이버의 I/O 스택 위치를 설정한다.
6. 두 드라이버는 IRP를 처리하고 요청된 I/O 작업을 완료하며, I/O Manager과 다른 시스템 구성 요소(위 그림에 표시되지 않음)가 제공하는 커널 모드 지원 루틴을 호출한다.
7. 드라이버들은 IRP를 I/O 관리자에게 반환하며, IRP 안의 I/O 상태 블록(I/O status block)을 설정하여 요청된 작업이 성공했는지, 또는 실패했다면 그 이유가 무엇인지를 나타낸다.
8. I/O Manager는 IRP에서 I/O 상태를 가져와 보호된 하위 시스템을 통해 원래 호출자에게 상태 정보를 반환할 수 있다.
9. I/O Manager는 완료된 IRP를 해제한다.
10. 만약 열기 작업이 성공했을 경우, I/O Manager는 하위 시스템에 파일 객체에 대한 핸들을 반환한다. 오류가 발생했을 경우에는 해당 오류에 대한 적절한 상태를 하위 시스템에게 반환한다.

<div class="bg"></div>

&nbsp;&nbsp; 하위 시스템이 데이터 파일, 장치, 또는 볼륨을 나타내는 파일 객체를 성공적으로 열면, 그 이후의 장치 I/O 작업 요청(보통 읽기, 쓰기, 또는 장치 I/O 제어 요청)에서 해당 파일 객체를 식별하기 위해 반환된 핸들을 사용한다. 이러한 요청을 수행하기 위해 하위 시스템은 I/O 시스템 서비스를 호출하며, I/O Manager는 이 요청들을 적절한 드라이버에게 IRP 형태로 전달한다.

# 2.4 I/O Stack Locations
&nbsp;&nbsp; I/O Manager는 생성한 IRP마다 하나의 I/O 스택 위치(I/O stack location)를 계층적으로 연결된 드라이버 체인 내의 각 드라이버에게 전달한다. 각 I/O 스택 위치는 [IO_STACK_LOCATION](https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/wdm/ns-wdm-_io_stack_location) 구조체로 구성되어 있다.

&nbsp;&nbsp; I/O Manager는 각 IRP에 대해 I/O 스택 위치들의 배열을 생성하며, 이 배열의 각 요소는 계층적으로 연결된 드라이버 체인의 각 드라이버에 해당한다. 각 드라이버는 이 배열에서 하나의 스택 위치를 소유하고 있으며, [IoGetCurrentIrpStackLocation](https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/wdm/nf-wdm-iogetcurrentirpstacklocation) 함수를 호출하여 해당 I/O 작업에 대한 드라이버 고유의 정보를 얻는다.

&nbsp;&nbsp; 이러한 드라이버 체인에서 각 드라이버는 [IoGetNextIrpStackLocation](https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/wdm/nf-wdm-iogetnextirpstacklocation)을 호출하여 다음 하위 드라이버의 I/O 스택 위치를 가져오고 이를 설정해야 한다. 또한, 상위 드라이버의 I/O 스택 위치는 작업에 대한 컨텍스트(context)를 저장하는 데에도 사용될 수 있으며, 이를 통해 해당 드라이버의 [*IoCompletion*](https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/wdm/nc-wdm-io_completion_routine) 루틴이 정리 작업(cleanup operations)을 수행할 수 있게 된다.

&nbsp;&nbsp; 파일 시스템 드라이버와 대용량 저장 장치 드라이버와 같은 두 개의 드라이버가 연쇄적으로 I/O 요청을 처리하는 경우, IRP에는 각 드라이버에 대응되는 I/O 스택 위치가 포함된다. 상위 드라이버가 하위 드라이버를 위해 IRP를 직접 생성하는 경우, 그 IRP에는 상위 드라이버 자신의 I/O 스택 위치는 포함되지 않으며, 하위 드라이버의 디바이스 객체에 설정된 StackSize 값을 기준으로
필요한 수의 I/O 스택 위치가 할당된다.
